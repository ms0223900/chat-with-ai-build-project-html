# 註解精簡掃描（Comment Trim）

## 目標

一個功能持續開發時（可能橫跨多次 `/feature`／`/fix`／`/adjust`／`/refactor` 呼叫），註解常常越寫越長、越寫越贅述——解釋顯而易見的 WHAT、引用當下任務/PR 脈絡、留下多段落說明——這些內容會拖累閱讀，卻不會被單一任務的視角注意到（每次都只看自己那一小塊）。本 skill 在**整個功能完成後**，針對這段期間累積的 git diff，掃描其中新增/修改的註解，直接精簡到只剩必要的 WHY。

**只動這次功能 diff 裡新增/修改的註解行，不動 diff 之外、這次功能沒碰過的既有註解。**

---

## 何時使用 / 何時不用

**使用時機**：

- 使用者說「這個功能做完了，註解太長/太多了，幫我整理一下」、「回頭檢查一下這次改動的註解」、「comment 精簡一下」。
- `next-task` 偵測到 epic/feature 收尾時，會建議使用者執行本 skill（見該 skill Step 8）。

**何時不用**：

- 單一任務剛完成、還沒累積多輪改動 → 直接在該任務自己的驗證步驟（`feature`/`fix`/`adjust`/`refactor` 各自 Step 已內建的自我檢查）處理即可，不需要為了一次小改動特地跑本 skill。
- 想找程式邏輯本身的重複/可簡化之處（非註解） → 用內建的 `simplify` skill。
- 想判斷是否該做架構重構 → 用 `/refactor-scan`，兩者是互補的獨立關注點，不要混在一起做。

## 執行流程

### Step 1：界定範圍

- 優先用「這次功能對應的 commit range」：偵測目前 branch 的基準分支（`main`/`master`/`develop`，依專案慣例），用 `git merge-base <base-branch> HEAD` 找出分岔點，掃描範圍＝分岔點到 `HEAD` 的整段 diff。
- 使用者明確指定其他 commit range 或檔案清單 → 依使用者指定為準。
- 找不到基準分支、也無法判斷範圍 → 退而求其次，改用尚未提交的 `git diff`／`git status`。
- 以上都無法決定範圍時，用一句話詢問使用者要掃描哪個範圍，不要自己亂猜。

### Step 2：抓出 diff 中新增/修改的註解行

依副檔名判斷註解語法，只看 diff 裡標記為新增/修改（`+`）的行：

| 副檔名 | 註解語法 |
|---|---|
| `.js` / `.ts` / `.jsx` / `.tsx` | `//`、`/* */`、JSDoc `/** */` |
| `.vue`（template 區塊） | `<!-- -->` |
| `.scss` / `.css` | `/* */` |
| 其他語言 | 依該語言慣例（`#`、`"""`/`'''` 等） |

### Step 3：逐條依精簡原則評估

**精簡原則**（統一標準，不要自行改寫出不同版本）：

- 預設不寫註解；只在「WHY 非顯而易見」時才留一句——隱藏的限制、不直覺的 invariant、針對特定 bug 的 workaround、會讓讀者意外的行為。
- 不解釋 WHAT（程式碼本身該做到讓人看懂在做什麼）。
- 不引用當下任務/修正/呼叫端（例如「用於 X 流程」「修正 issue #123 加的」、「這是給 next-task 呼叫時用的」）——這些內容屬於 PR 描述或 commit message，會隨程式演進而過期，不該留在程式碼裡。
- 不寫多段落 docstring 或多行註解區塊；一行說不清楚，通常代表該精簡的是程式碼本身（例如改個更清楚的命名），不是加更多註解解釋。

每條命中的註解標記為「贅述/過長，建議精簡」；其餘標記為「保留」。

### Step 4：直接精簡命中的註解

- 保留必要的 WHY，刪掉 WHAT 說明、任務/PR/呼叫端引用、多餘段落。
- 精簡後若整條註解已經沒有值得保留的 WHY → 直接整行刪除，不要為了「有東西」硬留一句空泛的話。
- **不確定是否有正當理由留長**（例如法規引用、複雜演算法背景本來就需要多行解釋、外部規格連結）→ **不要自己猜測後硬刪**，保留原樣，記錄到 Step 5 回報的「保留待確認」清單。
- 純文字編輯，不改動任何程式邏輯；改完不需要額外的功能驗證，但若不確定改動是否安全（例如註解緊鄰一段被註解掉的程式碼、拿不準是說明還是暫時停用的程式碼），一律保留不動並記錄待確認，不要順手清掉看起來像註解的程式碼。

### Step 5：回報

- 列出精簡了哪些檔案、各幾處（可附 1-2 個簡短前後對照範例，不需要每條都列）。
- 列出「保留待確認」清單：檔案位置＋為什麼不確定要不要精簡。
- 若掃描範圍內完全沒有命中贅述的註解 → 誠實回報「本次掃描未發現需要精簡的註解」，不要為了有產出硬找。

---

## Checklist

- [ ] 已確認掃描範圍（commit range 或 diff），範圍不明時已詢問使用者而非亂猜
- [ ] 只處理 diff 裡新增/修改的註解行，沒有動到這次功能沒碰過的既有註解
- [ ] 每條精簡/保留的判斷都對照同一套精簡原則，沒有各自發明標準
- [ ] 不確定是否該精簡的項目已保留原樣並列入待確認，沒有猜測後硬刪
- [ ] 沒有改動任何程式邏輯，純文字編輯
- [ ] 找不到需要精簡的內容時已誠實回報，沒有為了有產出硬找

## Examples

**「這個功能做完了，我發現註解越寫越長，幫我整理一下」**

→ Step 1：偵測目前在 `feature/SPRD-1234` branch，基準分支為 `main`，用 `git merge-base main HEAD` 抓出分岔點到 HEAD 的整段 diff。Step 2：抓出所有新增的 `//`／`/* */` 註解行。Step 3：發現多處註解在解釋「這個函式在做什麼」（WHAT）、或寫著「這是為了這次 SPRD-1234 的需求加的」（任務引用）→ 標記為贅述。Step 4：刪掉 WHAT 說明與任務引用，只留一處真正解釋隱藏限制的 WHY 註解。Step 5：回報精簡了 3 個檔案、共 6 處，附 1 個前後對照範例。

**「掃描後發現沒有需要精簡的註解」**

→ 這段期間新增的註解本來就精簡、只寫 WHY → 誠實回報「本次掃描未發現需要精簡的註解」，不做任何修改。
