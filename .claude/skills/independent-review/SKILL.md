# 獨立 Review Workflow

## 核心理念

寫東西的人很難自己抓出自己的盲點——即使局部每一步都邏輯正確，也常常漏掉「把整個系統當一個整體重新走一遍」這件事。本 skill 的解法：**另開全新、無主線對話記憶的 sub-agent**，讓它完全不知道「為什麼這樣改」「已經得出什麼結論」，強迫它自己從檔案、git diff、git log/commit 紀錄重建脈絡，並主動質疑、找反例——而不是順著已經寫好的敘述去確認「看起來合理」。

**本 skill 只負責審查與回報，不修改任何檔案**。找到的問題要不要修、怎麼修，交給使用者決定——依標的是產品程式碼還是 skill/文件本身，接續的動作不同，詳見文末「與其他 skill 的關係」。

---

## 何時使用 / 何時不用

**用**：使用者明確要求「獨立 review」「幫我批判性檢查一下」「找人重新看一遍」，通常發生在一段實作/修改**收尾後、宣告完成前**的最後一道關卡。

- 也適用於**尚未開始實作**的標的：`/ticket-to-ai-spec` 產出 spec 後會自動呼叫本 skill，此時審查標的是「剛產出的 spec 文件」而非「已完成的修改」——沒有 diff 可看，改成讓 sub-agent 帶著 spec 提到的 Impacted Areas/Technical Boundaries 去現有程式碼裡查證假設是否成立、有沒有既有 bug 會擋住之後的驗收（詳見 Step 0/Step 2 的對應說明）。

**不用（改用其他 skill）**：

| 情境 | 改用 |
|------|------|
| 只是核對 US 驗收條件是否已在程式碼中實現 | `us-acceptance-check`（機械式核對，非批判式挑錯） |
| 只是核對 PR 有沒有做到規格內的事、有沒有動到不該動的地方 | `pr-acceptance-checklist` |
| 一般程式碼 bug／簡化／效能審查 | 內建的 `code-review` |
| 安全性審查 | 內建的 `security-review` |

## 執行流程

### Step 0：判斷審查標的與範圍

1. 使用者有明確指定標的（檔案路徑、plan 路徑、「review 我剛剛的修改」）→ 直接使用。
2. 沒有明確指定 → 依序嘗試：`git status`/`git diff` 抓尚未 commit 的變更；若乾淨，改抓最近幾筆 commit（`git log -p -n 5` 或依對話語境判斷範圍）。仍無法判斷範圍時，用一句話詢問使用者要 review 什麼，不要自己亂猜範圍。
3. 收集背景素材——這些是要交給 sub-agent 的「第一手資料」，**不是主線自己的分析結論**：
   - 審查標的本身的 diff／內容
   - 相關 `git log`/commit message（理解「當初為什麼這樣做」，避免把刻意決策誤判成 bug）
   - 被引用或應保持一致的其他檔案（例如同一批修改觸及的其他檔案）
   - 對應的 US／spec／plan 文件（若有）

   若標的是**尚未實作的 spec/plan**（例如 `/ticket-to-ai-spec` 的產出）：沒有 diff 可抓，改成收集 spec 全文、原始 Ticket、spec 提到的 Impacted Areas 對應的現有程式碼路徑，作為 sub-agent 查證假設的第一手素材。

### Step 1：規模評估 — 決定開 1 個還是最多 3 個 sub-agent

| 情境 | 數量 |
|---|---|
| 小範圍、單一檔案、有既有 pattern 可對照 | 1 |
| 大規模重構 | 最多 3 |
| 沒有過往文件/pattern 可依循的調整 | 最多 3 |
| 相依性較多的修改（牽動多個檔案/模組） | 最多 3 |
| 涉及較複雜演算法邏輯的新功能（例如購物車計算） | 最多 3 |
| 使用者明確要求「多找幾個人看」 | 依使用者指定，上限 3 |

其餘情況預設 1 個就夠，不要為了「看起來慎重」而無腦開滿 3 個。

### Step 2：撰寫 sub-agent prompt 並平行送出

用 **Agent 工具**、`subagent_type: general-purpose`，需要平行審查時在**同一則訊息**中送出多個 Agent 呼叫。

不論開幾個，每個 sub-agent 的 prompt 都必須做到：

- **明確告知它是獨立審查者**：完全不知道主對話的討論脈絡、動機或已經得出的結論。只附上審查標的（路徑/diff/commit range）與 Step 0 收集到的第一手素材，**不要餵它主線的分析結論**，避免污染判斷。
- **明確要求唯讀**：只能讀取（`Read`/`Grep`/`Bash` 的 `git` 唯讀指令/`WebFetch`/`WebSearch`），不可修改、新增、刪除任何檔案。
- **指示具體的自我質疑做法**（開 3 個時，一個 agent 負責一個視角；只開 1 個時，同一個 agent 要涵蓋全部三個視角）：

  - **視角 A／邏輯與情境追蹤**：挑 1～2 個具體案例，在描述的流程或程式碼中逐步手動「跑一遍」，確認每一步真的走得通，有沒有死結、循環依賴或推論漏洞。
  - **視角 B／跨檔案與外部事實一致性**：核對所有被改動/引用的檔案之間有沒有互相矛盾（規則 vs 自己舉的範例、術語、依賴方向、前後步驟）；任何引用的外部 API/函式庫用法，要實際核對是否真的存在、是否符合目前版本慣例，不能因為「看起來合理」就假設是對的。
  - **視角 C／假設與邊界質疑**：主動列出這個改動假設了什麼，嘗試找反例；查 `git log`/commit message，判斷看起來奇怪的地方是否其實是刻意設計（避免誤判成 bug）；檢查有沒有漏掉的邊界情況、環境差異（本機 vs CI）、舊資料/舊格式相容性。

  審查標的是**尚未實作的 spec/plan**時，三個視角改為對照現有程式碼查證：spec 提到的模組/API/欄位是否真的存在、行為是否與 spec 假設相符；這個需求範圍內是否已有既有 bug、資料狀態或設計限制，會導致就算照 spec 實作完，AC 依然無法通過驗收；以及其他跟這次需求沒有直接依賴、但盤點過程中發現的問題。

- **要求誠實**：找不到問題就照實回報「沒發現嚴重問題」，不要為了顯得有產出硬掰；每個發現都要附具體證據（檔案路徑/行號、diff hunk、或具體反例情境），不能只給印象式評語。

### Step 3：彙整報告

**先做一次唯讀保證的事後驗證**：Step 2 的「唯讀」目前只靠 prompt 要求——`general-purpose` 這個 subagent type 本身仍有完整的 Edit/Write 權限，`Agent` 工具也沒有能限制單次呼叫工具集的參數，不是工具層強制。因此每個 sub-agent 回報後：

- 若審查標的在 git 倉庫內：跑一次 `git status`（若 Step 0 當下就已經有其他未 commit 的變更，需以 Step 0 記錄的快照為基準比對前後差異，而不是看到任何 pending change 就誤判）確認沒有新增/修改/刪除任何檔案。
- 若標的不在 git 倉庫或無法比對：至少對照 Step 0 收集的檔案清單，抽查關鍵檔案的內容有沒有被動過。
- 確認乾淨 → 正常進入下面的彙整。
- 若發現任何檔案被動過 → **不要自行復原或掩蓋**，明確告知使用者：是哪個 sub-agent、動了哪個檔案、目前實際 diff 內容為何，交由使用者決定要保留還是回退；並把這個違規本身也列進最終報告（視為 P0——這代表審查過程本身破壞了「只審查不修改」的核心承諾）。

- **只開 1 個 agent**：直接整理其回報即可。
- **開 3 個 agent**：主線讀取全部回報，去重合併：
  - 不同視角各自指向同一個問題的不同面向 → 合併成一條，標註「多視角共同指向」（信心較高）。
  - 兩個視角互相矛盾（例如一個判斷是 bug、一個判斷是刻意設計）→ 兩種說法都列出來，交給使用者定奪，**不要自己選邊**。
- 依嚴重度排序：P0（會直接導致流程卡死/產出錯誤結果）> P1（會誤導後續使用者/邏輯錯誤）> P2（潛在風險或不一致，但影響小）。
- 每個發現包含：位置（檔案/行號/diff hunk）、問題描述、為什麼有問題（具體情境，而非「感覺怪怪的」）、（可選）建議修正方向。
- **若有 sub-agent 沒有正常回報**（逾時、被使用者中途跳過、或回報內容明顯不完整/答非所問）：不要當作「那個視角沒發現問題」直接略過。在最終報告中明確標註哪個視角缺漏、涵蓋率因此打了折扣，並詢問使用者是否要單獨重跑那個視角，而不是靜默地用剩下 1～2 份報告冒充完整的 3 視角覆蓋。
- **不修改任何檔案**。結尾提示使用者：可自行決定是否接著處理——若標的是產品程式碼，接呼叫 `/fix`／`/feature`／`/refactor`／`/adjust`；若標的是 skill／文件本身（例如本次審查的對象就是 `.claude/skills/` 下的檔案），這些 skill 的守備範圍是產品程式碼，並不適用，應直接請 Claude 動手編輯對應文件，或若是要收斂一整批任務的經驗才用 `/distill-playbook`。

## Checklist

- [ ] 已明確判斷審查標的與範圍，沒有自己亂猜
- [ ] 已依規模評估決定 1 或最多 3 個 sub-agent
- [ ] 每個 sub-agent 都是全新、未被主線結論污染，且被明確要求唯讀
- [ ] 已用 `git status`（或檔案清單比對）驗證唯讀承諾確實有被遵守，而不是只信任 prompt
- [ ] 若開 3 個，三個視角分工明確、彼此沒有重複審查同一件事
- [ ] 找不到問題也誠實回報，沒有為了有產出硬掰
- [ ] 最終報告依嚴重度排序、每條發現都附具體證據
- [ ] 沒有修改任何檔案；已提示使用者後續可接哪個技能處理

## Examples

> 以下兩個範例純粹示範**流程怎麼走**，範例裡提到的具體發現（如「排程死結」「API 誤用」）只是虛構的示意，不是真的審查結論。**若本檔案本身也被當作素材交給 sub-agent**（例如審查標的包含 `independent-review/SKILL.md` 自己），要明確提醒該 sub-agent：這個 Examples 區塊是流程示範，不是既有結論，不要把它當成「已經有人發現過這些問題」而去確認或迴避。

**「幫我獨立 review 一下我剛剛改的這幾個 skill 文件」**

→ Step 0：`git diff` 抓出被改動的檔案清單與 diff、`git log` 看最近幾筆 commit message。Step 1：多檔案互相依賴（skill 之間互相引用），且沒有既有 pattern 可對照（TDD 欄位是新概念）→ 判定開 3 個。Step 2：三個 Agent 呼叫在同一則訊息平行送出，分別交付視角 A/B/C 的 prompt，只附 diff 與相關檔案路徑，不附「為什麼這樣改」的主線解讀。Step 3：彙整三份回報（示意，非真實案例）——例如視角 A 抓到排程死結、視角 B 抓到 API 誤用與範例矛盾，依嚴重度排序輸出，不修改任何檔案。

**「這個小改動幫我 review 一下」（單一檔案、小範圍格式調整）**

→ Step 1 判定範圍小、有既有慣例可對照 → 開 1 個 sub-agent，指示它涵蓋視角 A/B/C 全部三種質疑方式，回報整理後直接給使用者。

## 與其他 skill 的關係

- 本 skill 產出的報告如果確認需要修正，接續怎麼修**依標的類型而定**，本 skill 本身**不越界動手**：
  - 標的是**產品程式碼**（Vue/Nuxt/Next 等專案）→ 呼叫 `/fix`（工具鏈錯誤/邏輯錯誤修正）、`/feature`（功能性調整）、`/refactor`（重構）、`/adjust`（既有功能補充調整）。
  - 標的是 **skill／文件本身**（例如 `.claude/skills/` 下的 `SKILL.md`/`reference.md`，或一份 plan/spec 文件）→ `/fix`／`/feature`／`/adjust` 的守備範圍是產品程式碼，並不適用；直接請 Claude 動手編輯對應文件即可，若是要把一整批任務收尾後的經驗收斂進 Playbook/Skill/CLAUDE.md 才用 `/distill-playbook`。
- 與 `us-acceptance-check`/`pr-acceptance-checklist` 的差異：那兩個是「對照既有規格/AC 做機械式核對」，本 skill 是「不預設任何結論、主動質疑、找反例」的批判式審查，兩者可以互補而非取代。
