# 文件精簡掃描（Doc Trim）

## 目標

US／spec／playbook 這類文件會隨著多輪迭代（`ticket-to-ai-spec` 重新產出、`user-stories`／`adjust` 多次追加、`distill-playbook` 多次蒸餾）越寫越長、越來越囉唆——不是因為資訊真的變多，而是敘述性文字在每次修改時被重複鋪陳、填充語持續累積。這些檔案會被 `next-task`／`us-acceptance-check`／`distill-playbook` 等技能反覆完整讀取，越囉唆代表未來每一次讀取都要花更多 token。

本 skill 負責回頭掃描，把「已經確定要保留」的內容用更精簡的措辭重寫——**只精簡表達方式，不刪減任何實質資訊**，也**不動任何結構化／機器解析元素**。

---

## 何時使用 / 何時不用

**使用時機**：

- 使用者說「這份 spec/US 太囉唆了，幫我精簡」、「這個 playbook 太長了，整理一下」、「這個 epic 的文件都精簡一下，省 token」。
- `next-task` 偵測到 epic/feature 收尾時，會建議使用者執行本 skill（見該 skill Step 8）。

**何時不用**：

- 想精簡程式碼註解 → 用 `/comment-trim`（同樣的精簡精神，套用在程式碼上）。
- 想決定 Playbook 條目該不該保留、是否過期、是否與既有內容重複 → 那是 `/distill-playbook` 的職責（內容取捨），本 skill 只精簡「已經決定留下來的內容」怎麼表達；兩者可以先後執行（先 `/distill-playbook` 收斂內容，再用本 skill 精簡措辭），不要拿本 skill 代替 `/distill-playbook` 做取捨判斷。
- 文件本身還在頻繁修改中（例如任務還沒收尾、還會再迭代）→ 先不處理，避免精簡完馬上又被下一輪迭代覆蓋掉，白做工。

---

## 絕對不能動的部分（安全邊界，務必遵守）

精簡前，先辨識這些「機器解析用」的結構元素，逐字保留：

- **Checkbox 標記本身**：`[x]`／`[ ]`／`[⚠️]`／`[❌]`／`[🔍]`（標記旁邊的說明文字可以精簡，但標記符號本身不能動）
- **Heading 文字**，尤其是 `^#{2,4}\s*驗收說明`、`## 全域驗收 Checklist`、`## 重構掃描記錄` 等——`next-task`／`refactor-scan` 靠這些 heading 字串用 regex 判斷任務狀態
- **依賴鏈 ASCII 圖**（`─►`／`├─►`／`└─►`／`┬`／`┼` 等框線字元與其呈現的節點名稱、箭頭方向）
- **Watermark／水位線記錄**（例如「取材自 TASK-xxx ~ TASK-yyy」「已掃描至：TASK-004」）
- **AC（驗收條件）本身的判定語意**——可以讓句子更精簡，但不能改變它原本驗收的行為範圍
- **YAML frontmatter**（`name`／`description`）

不確定某段文字是不是機器解析的一部分時，保守起見**不要動**，寧可少精簡一點，也不要冒著弄壞其他 skill 自動化判斷的風險。

## 執行流程

### Step 1：界定範圍

- 使用者明確指定檔案／目錄 → 直接使用。
- 沒指定 → 依目前 branch 找 ticket key，套用 [next-task/reference.md](../next-task/reference.md) §一的追蹤目錄解析演算法找出對應的 `docs/user-stories/<EPIC>/`，一併納入同一個 ticket key 底下相關的 `docs/specs/`、`docs/playbooks/` 檔案。
- 範圍仍不明時，用一句話詢問使用者要精簡哪些檔案，不要自己亂猜。

### Step 2：分類每份文件裡的內容

逐檔讀取，把內容分成兩類：

- **結構化元素**：見上方「絕對不能動的部分」——原樣保留，不精簡。
- **敘述性文字**：Context/Problem/Goal 摘要、驗收說明裡的說明句、Playbook 條目的敘述、任何自由書寫的段落——這些才是精簡的對象。

### Step 3：依精簡原則改寫敘述性文字

- 不重複結構化欄位已經表達過的資訊（例如 Context 段落把 AC 已經寫過的內容整段複述一次）。
- 去除填充語（「基本上」「可以說是」「值得注意的是」「如上所述」）與不必要的鋪陳，直接講結論。
- 多句能濃縮成一句就濃縮，但不能因此遺漏語意——例外情況、條件限制、決策理由都要保留，只是講得更精煉。
- 已經很精簡的段落不要為了改而改。

### Step 4：不確定是否可以精簡時

若某段敘述牽涉還沒讀懂的業務語境，或精簡後可能弱化重要的例外/限制說明 → **不擅自精簡**，保留原樣，列入 Step 5 回報的「保留待確認」清單。

### Step 5：回報

- 列出處理了哪些檔案、精簡了哪些段落（可附 1-2 個前後對照，不需要每處都列）。
- 明確聲明「結構化元素（checkbox／heading／依賴圖／watermark）未被更動」。
- 列出保留待確認的部分（若有）。
- 若整批檔案本來就已經夠精簡 → 誠實回報「未發現需要精簡的敘述」，不要為了有產出硬改。

## Checklist

- [ ] 已確認掃描範圍，範圍不明時已詢問使用者而非亂猜
- [ ] 已明確區分「結構化元素」與「敘述性文字」，只精簡後者
- [ ] Checkbox 標記、heading 文字、依賴鏈 ASCII 圖、watermark 章節、AC 語意皆未被更動
- [ ] 精簡後沒有遺漏原本的資訊（例外情況、限制、決策理由都還在，只是措辭更精煉）
- [ ] 不確定是否該精簡的段落已保留並列入待確認，沒有擅自砍內容
- [ ] 找不到需要精簡的內容時已誠實回報，沒有為了有產出硬改

## Examples

**「這份 spec 的 Context 段落把驗收標準整段複述一遍」**

→ 把 Context 精簡成一句話摘要（Problem/Goal 各一句），驗收標準本身留在自己的章節不動，不重複。

**「playbook 條目歷經三次蒸餾後疊了三段幾乎重複的敘述」**

→ 濃縮成一段，保留三段裡各自不同的細節，去掉重複鋪陳；watermark 的「取材自」清單不動。

**「掃描後這批文件本來就精簡」**

→ 誠實回報「未發現需要精簡的敘述」，不做任何修改。

## 與其他 skill 的關係

- `distill-playbook` 負責「這個內容該不該留、會不會重複/過期」的取捨判斷；本 skill 只負責「已經決定留下來的內容，措辭能不能更精簡」。兩者互補，可以接續執行，不要互相取代。
- `comment-trim` 是同一套精簡精神，套用在程式碼註解上；本 skill 套用在 US/spec/playbook 等文件的敘述性文字上。
