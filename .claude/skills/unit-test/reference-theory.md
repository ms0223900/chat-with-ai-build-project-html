# Unit Test Reference — 理論篇（框架無關）

本文件收錄單元測試的基礎理論，適用於任何框架（Jest、Vitest、JUnit、pytest…）。單元測試是測試金字塔的地基，這份理論說明「什麼樣的單元測試才算好」。

---

## 一、FIRST 原則

- **Fast（快）**：跑起來要在毫秒等級；慢的測試會被跳過或降低執行頻率，拖慢回饋。
- **Independent（獨立）**：測試之間不依賴彼此的狀態或執行順序；單獨跑或任意順序跑都要能過。
- **Repeatable（可重複）**：任何環境、任何次數都跑出一樣結果（不依賴真實網路、真實時鐘、共用 DB 或亂數）。
- **Self-validating（自我驗證）**：通過/失敗由斷言/例外自動判定，不需要人工讀 console 輸出來判斷。
- **Timely（即時）**：在寫production code的當下或前後就寫測試，而不是事後補幾週。

---

## 二、Arrange-Act-Assert（AAA）結構

- **Arrange**：準備輸入、依賴與前置條件（建構被測物件、設定測試替身）。
- **Act**：執行「這個測試要驗證的那一個行為」（通常是一次方法/函式呼叫）。
- **Assert**：驗證可觀察的結果（回傳值、狀態變化、或某個互動確實發生）。
- 三段之間用空行或簡短註解分隔；如果一個測試沒辦法乾淨拆成 AAA 三段，通常代表它一次測了太多事。
- 測試內避免出現邏輯（迴圈、條件判斷）——有分支的測試等於「未被測試的程式碼」。

---

## 三、測試替身分類（Test Double Taxonomy，Meszaros）

| 類型 | 定義 | 使用時機 |
|------|------|----------|
| Dummy | 只為了滿足參數簽章而傳入，實際不會被使用 | 佔位用（例如一個永遠不會被呼叫的 logger） |
| Stub | 對測試中的呼叫回傳固定/預先設定好的回應 | 控制間接輸入 |
| Spy | 會記錄呼叫方式（參數、次數）的 stub，供測試事後斷言 | 需要驗證「有沒有被這樣呼叫」但不阻斷流程時 |
| Mock | 預先設定「應該收到哪些呼叫」的期待，沒發生就讓測試失敗 | 驗證關鍵副作用確實發生（例如「有寄出信件」） |
| Fake | 可運作但簡化過的實作（例如記憶體版 DB） | 足夠給測試用，但不能上 production |

- **選用原則**：控制輸入、做 state-based 斷言時優先用 stub/fake；只有在需要驗證「某個有副作用的關鍵互動確實發生」時才用 mock。
- **過度 mock 的警訊**：把被測單元觸及的每樣東西都 mock 掉（包含簡單的 value object、語言自帶的 collection）、mock 了你不擁有的型別、一個測試裡 `verify()`/`toHaveBeenCalled()` 比真正的斷言還多。過度 mock 會讓測試變成「驗證程式碼呼叫了 X」而非「驗證程式碼行為正確」，導致保留行為的重構也會誤報壞掉。

---

## 四、測試金字塔

- **單元測試**（底層、數量最多）：單一函式/類別，隔離依賴（用替身），快、能精準定位失敗點。
- **整合測試**（中層）：多個真實模組/協作者接在一起（例如 service + 真實 DB），能抓到單元測試漏掉的介接/契約問題。
- **端到端測試**（頂層、數量最少）：透過真實使用者介面跑完整系統，慢、較脆弱，但能驗證整條流程真的能動。
- 經驗法則：盡量把驗證下推到金字塔底層——更便宜、更快、失敗訊號更精準；越上層的測試數量應該越少。

---

## 五、測試行為，而非實作

- 好的測試斷言**公開 API／可觀察的結果**（回傳值、拋出的錯誤、呼叫端能看到的狀態、發出的事件）——不斷言私有欄位、內部呼叫順序或 helper 方法本身有沒有被呼叫。
- **測試被實作綁死的警訊**：
  - 重新命名一個私有方法或調整內部步驟順序（輸出沒變），測試卻壞了。
  - 斷言了呼叫端看不到的內部狀態（需要用反射或特地為了測試暴露內部細節）。
  - mock 掉一個協作者，並斷言 mock 被呼叫時傳入的確切內部參數，而不是檢查最終效果。
  - 重構（行為不變、結構改變）卻逼得你要改好幾個測試。
- 試金石：如果你能把實作整個重寫一遍、輸入輸出完全不變，卻完全不用動這個測試——那它就是在測行為；反之就是在測實作。

---

## 六、一個測試只驗證一個概念

- 一個測試 = 一個邏輯行為/情境，不是「只能有一個 `assert()`」——同一個結果的多個欄位在同一個測試裡一起斷言完全沒問題。
- 不同情境（happy path、每個邊界值、每個錯誤狀況）分開寫成不同測試，這樣失敗時的測試名稱就能直接告訴你壞在哪裡。
- 如果描述一個測試需要用到「而且」、「或者」，通常代表它其實是兩個測試。

---

## 七、命名慣例

- 描述**行為 + 條件**，而不是方法名稱：例如 `should return empty list when input is null`，或 `shouldThrowError_whenAmountIsNegative`。
- Given-When-Then 風格也適用：`given negative balance, when withdraw is called, then it throws InsufficientFundsError`。
- 好的名稱讓人不用打開測試內容，光看失敗報告就知道壞了什麼。

---

## 八、Mutation Testing（自我驗證）

- 工具會刻意「突變」production code（把 `<` 改成 `<=`、刪掉一行、反轉一個條件），然後重跑測試。
- 如果測試在突變後的程式碼上仍然通過，代表它其實沒有真正驗證到那段邏輯——這是 line/branch coverage 這類指標看不出來的覆蓋率破洞。
- 因為執行較慢，建議當作**定期審視測試強度的稽核手段**，而非每次 commit 都要跑的關卡。
