# 蒸餾 Playbook（巨蒸餾）

本 skill 用於**一個 feature / refactor / bug-fix / epic 收尾時**，把過程中累積的「即時蒸餾」筆記（散落在各 TASK/US 文件的「驗收說明」）收斂進可複用的知識載體。這是「巨蒸餾」——跟 feature/refactor skill 做完單一任務後的「即時蒸餾」是互補關係，不是取代：即時蒸餾負責「當下寫下來」，巨蒸餾負責「定期收斂、去重、防止過期」。

---

## 何時使用 / 何時不用

**使用時機**：

- 使用者明確要求整理/蒸餾某個 feature、epic、ticket 的經驗
- 一系列 TASK / user story（例如整個 SPRD-XXX、docs/user-stories/<EPIC>/ 底下的所有任務）全部完成，準備收尾

**不適用**：

- 單一 TASK/US 剛完成 → 那是即時蒸餾，直接寫在該任務自己的「驗收說明」，不必呼叫本 skill
- 想知道某個 US 的驗收條件是否通過 → 用 `us-acceptance-check`
- 想知道某次重構邏輯有沒有跑掉 → 用 `refactor-review`

---

## 執行流程

### Step 1：確認範圍與既有載體

1. 找出這個 feature/epic 的 TASK/US 文件目錄（例如 `docs/user-stories/<EPIC>/`）與其索引/README（若有）。
2. 找出是否已有對應的 Playbook（`docs/playbooks/*.md`）或相關 Skill（`.claude/skills/*/SKILL.md`）。
   - 若已有 Playbook：讀取其「附錄」或開頭的**產出依據 / watermark**（記錄了「取材自 TASK-xxx ～ TASK-yyy」），藉此判斷哪些 TASK 的經驗**已經蒸餾過**，避免重複勞動。
   - 若沒有 Playbook：本次視為第一次蒸餾，建立新檔案（位置與命名參考 `docs/playbooks/<EPIC>-xxx-playbook.md`，或若專案有既有慣例則沿用）。
3. 若找不到明確範圍（使用者沒指定是哪個 epic），先詢問使用者要蒸餾哪個 feature/epic，不要自行猜測。

### Step 2：收集素材

- 讀取範圍內、且**尚未蒸餾過**的每個 TASK/US 文件的「驗收說明」（或等效區塊）。
- 針對每份文件，摘出：決策與理由（尤其是「為什麼留 / 為什麼移」「查證後發現不需要調整」這類明確決策）、踩過的坑、可複用的程式片段或指令、與既有假設不符的發現。
- 忽略純粹的「測試通過」「AC 已勾選」這類例行結論——只收集**非顯而易見**的內容（跟 `/memory` 的收錄原則一致）。

### Step 3：分類每條素材

對每條收集到的素材，判斷該歸屬哪個載體：

| 素材性質 | 歸屬 | 範例 |
| --- | --- | --- |
| 可重複觸發的「怎麼做」程序 | **Skill**（`.claude/skills/`） | 「整合測試骨架該怎麼寫」 |
| 這個專案/領域特有的「學到什麼」，需要人主動查閱 | **Playbook**（`docs/playbooks/`） | 「FinalGameList 為什麼留在原元件」 |
| 少量、永遠成立、極簡短的規則 | **CLAUDE.md** | 「一律用 rtk 前綴」 |
| 已經在目標載體裡寫過、或只是常規操作 | **捨棄** | 「測試通過」 |

一條素材可能同時適用多個載體時，選**最貼近未來使用情境**的那個——會被主動查閱的知識放 Playbook，會被自動比對觸發的程序放 Skill。

### Step 4：合併，不要盲目附加

修改目標檔案前，**先讀過整份既有內容**：

- 若新素材是既有條目的補充 → 更新該條目，不要在檔案尾端重複開一條類似的
- 若新素材推翻或修正了既有條目（例如後來擴充了某個 helper，原本記錄的限制不再成立）→ 直接改寫該條目，並可加註「已於 TASK-xxx 解除此限制」
- 若新素材與既有條目無關 → 依原檔案的分節慣例，加進最貼切的章節（不要無腦全部塞進「已知陷阱」或全部塞在檔尾）
- **多輪蒸餾下來，同一個條目容易疊出好幾段幾乎重複的敘述**（每次都在舊的說法上再補一段，而不是取代）；合併時順手把這類條目濃縮成一段、保留各段不同的細節，不要照單全收地並排堆疊。若整份檔案的措辭已經明顯囉唆、超出這次新素材觸及的範圍，提醒使用者可另外呼叫 `/doc-trim` 做一次全面的措辭精簡，不需要在本 Step 順手做全文改寫。

### Step 5：處理易過期的內容

蒸餾時順手檢查目標檔案裡是否有這類內容：

- **具體數字**（測試數量、行數、日期）：重新驗證後更新，或改寫成不依賴精確數字的敘述（例如「已有廣泛整合測試覆蓋」取代「199 個測試」），除非該數字本身就是這次要記錄的重點
- **已被後續變更取代的限制或做法**：依 Step 4 的規則改寫或移除
- 這一步不需要每次都全文審查整份文件，但至少要檢查**這次新素材直接觸及的段落**是否連帶過期

### Step 6：更新 watermark / 產出依據

在目標 Playbook（或 Skill）文件的「附錄」或開頭區塊，更新「取材自」清單，涵蓋本次新收錄的 TASK/US 編號，讓下次蒸餾能正確判斷起點。

### Step 7：回報

用清單列出本次蒸餾的結果，分「新增」「更新（含原因）」「移除/改寫（含原因）」三類，讓人可以快速核對、決定是否要調整。**不需要每條都先問過使用者才動手**——蒸餾判斷本身就是本 skill 的職責，但務必在回報中讓移除/改寫的部分清楚可見，方便事後修正。

---

## 與 feature / refactor skill 的關係

- `feature`、`refactor` skill 在完成單一任務後，會把非顯而易見的經驗直接寫進該任務自己的 US/TASK 文件「驗收說明」（即時蒸餾），**不需要**也**不應該**在那個當下就去動 Playbook/Skill——那樣會導致 Playbook 被瑣碎、未經整理的內容淹沒。
- 本 skill 只在 feature/epic **整個收尾**時執行一次，一次性把該範圍內累積的即時蒸餾筆記做整併、去重、防過期處理。
